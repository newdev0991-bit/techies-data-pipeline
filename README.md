# techies-data-pipeline

Central ingestion + deduplication pipeline that makes one Postgres database
(`myDatabase`) the **single source of truth** for NFULL and MFULL leads. Both
lead-source backends push their records here. NFULL is the authoritative base:
all NFULL source-row occurrences are retained. MFULL is compared by normalized
phone only; an MFULL phone already present in NFULL is not copied, and repeated
MFULL rows with the same phone keep one combined lead. The read API serves the resulting
NFULL + MFULL-only set to both frontends instead of raw Google Sheets.

MFULL deletes its live source rows after 24 hours. The dedicated
`mfull-capture` GitHub Action snapshots it every 15 minutes (with three retries)
into Postgres. Ingestion is append-only, so source deletion does not remove the
captured raw rows or combined leads.

**Milestone 1** (no AI, no cost) is the ingestion + dedup + read API above.
**Milestone 2** adds a background validation worker that reuses the existing
`techies-validator-backend` (`/fetch-results` + `/analyze`) to classify leads
APPROVED / REVIEW_REQUIRED / REJECTED — after cheap deterministic pre-checks.

## Local development

```bash
cp .env.example .env          # dev defaults already point at the docker DB
docker compose up -d          # local Postgres on host port 5433
npm install
npm run migrate               # create tables
npm run ingest:now            # one-shot NFULL-first + MFULL-only backfill
npm run check:contract        # pure data-contract unit checks (no DB)
npm run smoke                 # end-to-end verification (synthetic data, no cost)
npm run dev                   # start the web service on PORT (default 4100)
```

`npm run smoke` proves the Milestone-1 completion conditions: idempotent
re-ingest creates no extra source occurrences, repeated NFULL rows remain,
and MFULL-vs-NFULL/within-MFULL matching uses normalized phone only.

## Endpoints

Internal (auth: `Authorization: Bearer $PIPELINE_API_TOKEN`):

| Method | Path | Purpose |
| --- | --- | --- |
| POST | `/internal/ingest/nfull` | Ingest NFULL rows `{ rows: [ {..verbose headers..} ] }` |
| POST | `/internal/ingest/mfull` | Ingest MFULL rows |
| POST | `/internal/ingest/batch` | Mixed batch `{ rows: [ { source, row } ] }` (cron) |
| POST | `/internal/tick` | Scheduled capture; accepts optional `sources: ["NFULL","MFULL"]` |
| GET | `/internal/pipeline/status` | Per-source last run + totals |
| GET | `/internal/pipeline/runs` | Recent import runs |

Public (auth: `X-API-Key: $PUBLIC_API_KEY`; called server-side by the frontend proxies):

| Method | Path | Purpose |
| --- | --- | --- |
| GET | `/api/leads?source=NFULL\|MFULL\|BOTH&date=&exported=&page=&pageSize=&format=json\|csv` | Deduplicated leads with a `source` badge |
| GET | `/api/stats` | Overview + source-comparison counts |
| POST | `/api/exports` | Mark leads exported `{ master_ids: [], context? }` |
| GET | `/health` | Liveness + DB check |

Idempotency: send `Idempotency-Key` (or body `idempotencyKey`) on ingest calls;
re-sending the same key returns the stored counts without reprocessing.

## Existing-database cutover

Migration `005_phone_precedence.sql` retains masters created by the retired
global URL/phone policy for audit, but hides them from the combined read model.
Backfill the corrected records before deploying the new read API so there is no
empty-output window:

```bash
# With the production DATABASE_URL and Google source variables loaded:
npm run migrate
npm run ingest:now
```

Then deploy the web service and run `npm run shadow-compare`. The combined count
should be every NFULL row plus one MFULL-only row per normalized phone. Legacy
masters and their raw records stay stored, but are excluded from output and
current source-count comparisons.

Enable both GitHub Actions workflows and configure the repository secrets
`PIPELINE_BASE_URL` and `PIPELINE_API_TOKEN`. A failed `mfull-capture` run means
the 24-hour source-retention safety net is not healthy and should be investigated
before the oldest uncaptured rows expire.

## Deploy (Render)

`render.yaml` provisions the web service; the database is supplied through
`DATABASE_URL`. Render's free plan has no pre-deploy command, so run migrations
manually after schema changes. `PIPELINE_API_TOKEN` and `PUBLIC_API_KEY` are
generated by Render — copy the token to the scheduler and the key into the two
frontend Vercel projects.
