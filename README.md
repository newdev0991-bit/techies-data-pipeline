# techies-data-pipeline

Central ingestion + deduplication pipeline that makes one Postgres database
(`myDatabase`) the **single source of truth** for NFULL and MFULL leads. Both
lead-source backends push their records here; the pipeline normalizes and
deduplicates them (a lead found by both sources becomes one `master_lead` with
two `lead_sources` rows) and serves them from one read API that both frontends
consume instead of raw Google Sheets.

**Milestone 1** (no AI, no cost) is the ingestion + dedup + read API above.
**Milestone 2** adds a background validation worker that reuses the existing
`techies-validator-backend` (`/fetch-results` + `/analyze`) to classify leads
APPROVED / REVIEW_REQUIRED / REJECTED — after cheap deterministic pre-checks.

## Local development

```bash
cp .env.example .env          # dev defaults already point at the docker DB
docker compose up -d          # local Postgres on host port 5433
npm install
npm run migrate               # create tables
npm run check:contract        # pure data-contract unit checks (no DB)
npm run smoke                 # end-to-end verification (synthetic data, no cost)
npm run dev                   # start the web service on PORT (default 4100)
```

`npm run smoke` proves the Milestone-1 completion conditions: idempotent
re-ingest creates no duplicate master leads, and the same lead from NFULL + MFULL
converges to one master with two sources.

## Endpoints

Internal (auth: `Authorization: Bearer $PIPELINE_API_TOKEN`):

| Method | Path | Purpose |
| --- | --- | --- |
| POST | `/internal/ingest/nfull` | Ingest NFULL rows `{ rows: [ {..verbose headers..} ] }` |
| POST | `/internal/ingest/mfull` | Ingest MFULL rows |
| POST | `/internal/ingest/batch` | Mixed batch `{ rows: [ { source, row } ] }` (cron) |
| GET | `/internal/pipeline/status` | Per-source last run + totals |
| GET | `/internal/pipeline/runs` | Recent import runs |

Public (auth: `X-API-Key: $PUBLIC_API_KEY`; called server-side by the frontend proxies):

| Method | Path | Purpose |
| --- | --- | --- |
| GET | `/api/leads?source=NFULL\|MFULL\|BOTH&date=&exported=&page=&pageSize=&format=json\|csv` | Deduplicated leads with a `source` badge |
| GET | `/api/stats` | Overview + source-comparison counts |
| POST | `/api/exports` | Mark leads exported `{ master_ids: [], context? }` |
| GET | `/health` | Liveness + DB check |

Idempotency: send `Idempotency-Key` (or body `idempotencyKey`) on ingest calls;
re-sending the same key returns the stored counts without reprocessing.

## Deploy (Render)

`render.yaml` provisions a managed Postgres, the web service (runs `npm run
migrate` on pre-deploy), and the reconciliation cron. `PIPELINE_API_TOKEN` and
`PUBLIC_API_KEY` are generated by Render — copy the token into the two Flask
backends and the key into the two frontend Vercel projects.
